# 医美 H5 前后对比图项目 — 设计文档

- 日期：2026-06-11
- 状态：设计待审阅

## 1. 项目概述

一个面向手机移动端的医美 H5 应用。用户上传一张本人照片并录入文字描述（提示词），提交给后端；内部处理人员在网页管理后台看到原图和提示词后，手动完成修图工作，再上传处理后的结果图；用户端轮询到结果后，看到处理前后两张图片的滑块对比效果。

整个系统由三部分组成：

- **用户端 H5**（手机浏览器）：上传照片 + 文字 → 等待 → 看对比图 → 看历史
- **管理后台**（电脑浏览器）：登录 → 看待处理任务列表 → 看原图和提示词 → 上传结果图 → 设置图片留存天数
- **Node 后端服务**：提供 API、存储任务数据与图片、定时清理过期图片

## 2. 核心需求（已确认）

| 项目 | 决策 |
|------|------|
| 结果回传方式 | 同设备等待 + 简单定时轮询（每 3-5 秒），无需登录 |
| 用户身份 | 无账号，用浏览器 localStorage 记住本设备任务 |
| 提示词形式 | 自由文本 |
| 照片数量 | 单张进、单张出 |
| 对比图样式 | 滑块对比（前后两图叠加，拖动中间滑块看变化） |
| 处理时长 | 几分钟级，用户在页面等待 |
| 后台入口 | 独立网页管理后台（电脑端），多账号登录 |
| 任务流程 | 只走成功路径（不做驳回功能） |
| 用户历史 | 本设备可看自己提交过的所有任务列表 |
| 图片留存 | 按天自动清理，全局一个留存天数，管理员后台可设置 |

## 3. 技术栈

- **前端**：React + Vite（用户端 H5 与管理后台各为独立应用）
- **后端**：Node.js（Express 或 Fastify）
- **数据库**：SQLite（单文件，零外部依赖）
- **图片存储**：本地磁盘
- **代码组织**：单仓库 monorepo，三个独立包

## 4. 项目结构

```
医美h5/
├── packages/
│   ├── shared/          # 共享的 TypeScript 类型定义（任务状态、API 契约）
│   ├── user-h5/         # 用户端 H5（React + Vite，移动端优先）
│   ├── admin/           # 管理后台（React + Vite，桌面端）
│   └── server/          # Node.js 后端
│       ├── data/
│       │   ├── app.db   # SQLite 数据库文件
│       │   └── uploads/ # 图片存储目录
│       └── src/
├── docs/
└── package.json         # workspace 根配置
```

三个前端/后端包独立构建、独立部署。`shared` 包存放跨端共用的类型，保证前后端 API 契约一致。

## 5. 数据模型

### tasks 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (UUID) | 任务唯一标识，也用于轮询 |
| device_id | TEXT | 用户设备标识（前端生成并存 localStorage），用于查历史 |
| prompt | TEXT | 用户录入的文字提示词 |
| original_image | TEXT \| NULL | 原图文件路径（相对 uploads/）；清理后置空 |
| result_image | TEXT \| NULL | 结果图文件路径，未处理时为空；清理后置空 |
| status | TEXT | `pending`（待处理）/ `done`（已完成）/ `expired`（图片已清理） |
| created_at | INTEGER | 提交时间（Unix 毫秒） |
| completed_at | INTEGER \| NULL | 完成时间 |
| handled_by | TEXT \| NULL | 处理该任务的后台账号 |

> 状态说明：清理任务后将其 `status` 置为 `expired` 并把两个图片字段置空。前端据此区分"待处理 / 已完成 / 已过期清理"三种情况，避免与 `pending` 混淆。

### admins 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 自增主键 |
| username | TEXT | 登录用户名（唯一） |
| password_hash | TEXT | 密码哈希（bcrypt） |
| created_at | INTEGER | 创建时间 |

### settings 表（键值表）

| 字段 | 类型 | 说明 |
|------|------|------|
| key | TEXT | 设置项键名 |
| value | TEXT | 设置项值 |

初始包含 `retention_days`（图片留存天数，默认如 30）。

## 6. API 设计

### 用户端 API（无需登录）

- `POST /api/tasks`
  - 请求：`multipart/form-data`，字段 `image`（文件）、`prompt`（文本）、`deviceId`（文本）
  - 响应：`{ id, status: "pending", createdAt }`
- `GET /api/tasks/:id`
  - 轮询用。响应：`{ id, status, prompt, originalUrl, resultUrl, createdAt, completedAt }`
- `GET /api/tasks?deviceId=xxx`
  - 查本设备历史列表。响应：任务摘要数组（按时间倒序）

### 管理后台 API（需登录，携带会话凭证）

- `POST /api/admin/login` — 请求 `{ username, password }`，成功返回会话凭证（如 httpOnly cookie + token）
- `POST /api/admin/logout`
- `GET /api/admin/tasks?status=pending` — 任务列表，支持按状态筛选
- `GET /api/admin/tasks/:id` — 单任务详情（原图、提示词）
- `POST /api/admin/tasks/:id/result` — 上传结果图（`multipart/form-data`），将状态置为 `done`，记录 `handled_by` 和 `completed_at`
- `GET /api/admin/settings` / `PUT /api/admin/settings` — 读写留存天数

### 图片访问

- `GET /uploads/:filename` — 返回图片文件。原图路径不可被轻易枚举（文件名用随机/UUID）。

## 7. 数据流

### 提交与等待（用户端）

```
用户选照片 + 填提示词
   → 前端校验（有图、有文字、大小/格式）
   → 生成 deviceId（若无，存 localStorage）
   → POST /api/tasks（上传图片+文字+deviceId）
   → 后端存图、写 tasks 记录（status=pending）、返回 task.id
   → 前端把 task.id 存入 localStorage 历史
   → 进入等待页，每 3-5 秒 GET /api/tasks/:id
   → status 变为 done → 拉取 resultUrl → 显示滑块对比图
```

### 处理（管理后台）

```
后台人员登录
   → 看 pending 任务列表
   → 点开任务，看原图 + 提示词
   → 线下手动修图
   → 上传结果图 → POST .../result
   → 后端存结果图、status=done、记录处理人和完成时间
```

### 自动清理（后端定时任务）

```
后端定时器（如每天跑一次）
   → 读 settings.retention_days = N
   → 找出 created_at 早于 (now - N 天) 的任务
   → 删除其 original_image / result_image 文件
   → 将任务 status 置为 expired、两个图片字段置空（保留任务记录用于历史展示）
```

## 8. 关键界面

### 用户端 H5（移动端优先）

1. **上传页**：照片选择/拍照入口（带预览）、提示词文本框、提交按钮。
2. **等待页**：上传的原图缩略图、"处理中"动画、文案提示（几分钟，可留在本页）。轮询中。
3. **结果页**：滑块对比组件（前后两图叠加，拖动中间手柄）、保存/分享入口（可选）。
4. **历史列表**：本设备所有任务，每条显示缩略图、提示词摘要、状态、时间；点击进入对应等待页或结果页。

### 管理后台（桌面端）

1. **登录页**：用户名 + 密码。
2. **任务列表页**：默认显示待处理任务，可切到已完成；每行显示原图缩略图、提示词摘要、提交时间、状态。
3. **任务详情/处理页**：大图看原图、完整提示词、结果图上传区。
4. **设置页**：图片留存天数输入框。

## 9. 错误处理

- **上传失败**（网络/格式/超大）：前端给出明确提示，允许重试，不产生半截任务。
- **图片格式与大小**：前端限制常见图片格式（jpg/png/webp）和大小上限（如 10MB），后端二次校验。
- **轮询中网络中断**：轮询失败自动重试，不丢失 task.id（已存 localStorage），恢复后继续。
- **任务不存在 / 已清理**：用户访问已被清理的旧任务时，结果页显示"图片已过期清理"友好提示。
- **后台未授权访问**：管理 API 无有效会话一律返回 401，前端跳登录页。
- **结果图上传幂等**：同一任务重复上传结果，以最后一次为准。

## 10. 安全考量

- **管理后台鉴权**：密码 bcrypt 哈希存储；会话用 httpOnly cookie，防 XSS 窃取。
- **图片隐私**：用户照片为脸部敏感信息。文件名随机化避免枚举；按留存天数自动清理；管理 API 全程鉴权，普通用户无法访问他人任务（仅凭 task.id 直链，id 为不可猜测的 UUID）。
- **上传校验**：服务端校验文件类型与大小，防止伪装文件、超大文件攻击。
- **越权防护**：用户端 `GET /api/tasks/:id` 仅靠 UUID 不可枚举；如需更强隔离，可校验 deviceId 与任务归属（设计预留）。

## 11. 范围与非目标（YAGNI）

明确**不做**：

- 用户账号系统、登录、找回密码
- 微信登录 / 消息推送通知
- 多张照片、参考图上传
- 任务驳回 / 重做流程
- 实时 WebSocket 推送
- AI 自动修图（结果由人工上传）
- 支付、预约、客服等医美业务功能

这些都不在本期范围内，保持系统聚焦于"上传 → 人工处理 → 对比图"这一条主线。

## 12. 部署形态

单台服务器即可运行：一个 Node 进程提供 API 与图片服务，SQLite 文件 + uploads 目录挂在本地磁盘。两个前端构建为静态资源，可由同一 Node 服务托管或单独的静态服务器/Nginx 提供。无外部数据库或对象存储依赖，便于快速上线，后续如需扩展可平滑迁移到云数据库 + 对象存储。
